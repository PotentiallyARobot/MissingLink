What is PromptFlow™

A language, a file format, and the system that runs on them.

PromptFlow™ is a programming language for prompting neural networks. A PromptFlow™ process is an executable program that describes a sequence of model invocations — its inputs, its parameters, the chain of steps that feed one model’s output into the next, and the shape of the result.

Processes are stored as .promptflow files: electronic data files featuring programming data in the field of AI. A .promptflow bundle is portable, inspectable, and reproducible — the same file run twice against the same model produces the same plan.

The PromptFlow™ software platform is a complete system for authoring, executing, retrieving, and composing these processes. It runs on the web at missinglink.build/promptflow.

The three pools

Every process is system, private, or public.

A PromptFlow™ process always lives in one of three pools. The system pool holds the OS itself. The private pool holds your work. The public pool holds the shared graph of everyone else’s. The planner retrieves across the system and public pools; private processes are accessed directly by their owner.

System

The OS, written in itself.

System PromptFlow™ processes perform the essential functions of the operating system. The OS is built from inference: when a new neural network model releases, its system processes can be regenerated against the new model. The functions persist; their implementations adapt.

adaptation · optimization · consolidation · validation · evaluation · composition · retrieval · search
Private

Your processes, or theirs.

Private PromptFlow™ processes are owned. They belong to the user who created or forked them, and they remain invisible to everyone else until that user chooses to publish. Editing, monetization, deletion — the owner decides.

drafts · forks · in-progress work · monetized processes · team-internal recipes
Public

A shared graph of many authors.

Public PromptFlow™ processes form a graph. Alice’s storyboard generator composes Bob’s music recipe, which forks Carol’s tempo classifier. Authors weave through one another. Lineage, composition, and forks span the whole network.

published processes · forks · composed pipelines · community-authored recipes
Retrieval

The planner walks the system and the public graph.

When a user describes what they want in natural language, the planner — itself a system process — searches the system pool and the public graph and either selects an existing process or composes a new one from the parts it finds. Retrieval is how PromptFlow™ resolves intent into a program.

Private processes are not part of retrieval. They’re owned space, not searched namespace. A user invokes private processes directly — opening them from their workspace, running them by name, forking from them deliberately. The planner doesn’t reach into anyone’s private workspace; the user does.

This is the same distinction a conventional operating system draws between system paths that get searched (the kernel, shared libraries, the public package registry) and a user’s home directory, which gets accessed explicitly. The boundary protects ownership while keeping the searched namespace coherent.

Why “Operating System”

A specific claim, not a metaphor.

An operating system, by the technical definition, abstracts a hardware substrate into addressable programs, manages their execution, mediates between them through shared structure, and exposes a way to invoke them. PromptFlow™ does the same with neural networks as the substrate.

processes = programs
A .promptflow file is an executable description of work to be done by one or more neural networks. Inputs, parameters, ordering, output shape. A program.
retrieval = process lookup
A natural-language request resolves to a specific process by searching the system pool and the public graph — the searched namespace. Same role as a path lookup in a filesystem-based OS, just with semantic matching across two scopes instead of path resolution.
system pool = the kernel
The functions every other process depends on — adaptation, validation, composition, search — live in the system pool. They are themselves PromptFlow™ processes, so the kernel is in the same language as everything that runs on it.
composition = linking
One process invokes another by name. The result is a graph of authors’ work, linked through a single shared namespace. Public processes can call public processes; private processes can call public ones; system primitives underlie all of it.
forking = copy-on-write
When a user refines a system or public process, the system creates a private fork. The original is preserved. The fork can diverge freely. Both records remain in the graph, related by lineage.
generative = self-rebuilding
The OS’s own functions are written in the same language they operate on. When a new neural network model releases, those system processes can be regenerated against it. The kernel rewrites itself. That’s what generative means here.
